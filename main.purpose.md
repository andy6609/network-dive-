# 프로젝트 목적 및 배경 — Local Proxy 기반 네트워크 트래픽 분석 PoC

---

## 웹사이트 목적 & 스타일

- **용도**: 팀 발표용 소개 페이지 + 개인 포트폴리오 겸용
- **톤**: 광고 수준의 전문성 — 단순 설명 페이지가 아니라 제품 런칭 사이트처럼
- **무드**: 다크 테크 / 보안 느낌 / 고급스러운 미니멀. Apple 제품 소개 페이지나 사이버보안 기업 랜딩처럼 임팩트 있게
- **핵심 인상**: 보는 사람이 "이걸 직접 만들었다고?" 하는 느낌. 기술력과 완성도를 동시에 보여줌
- **스크롤 연동 루프 영상**: 배경에 Seedance 생성 영상을 깔고 스크롤에 따라 재생/오버레이 변화

---

## 한 줄 요약

사내 직원이 AI 서비스(ChatGPT, Claude 등)로 회사 파일을 업로드할 때, 이를 **Local Proxy로 가로채어 식별·분석**하는 시스템.

---

## 왜 필요한가 (프로젝트 배경)

브라우저 안에서 벌어지는 파일 업로드는 눈에 보이지 않고, HTTPS로 암호화되어 있어 네트워크에서도 그냥 지나간다.  
사내 직원이 회사 문서를 ChatGPT 같은 외부 AI 서비스에 업로드해도, 회사 입장에서는 이를 탐지할 수단이 없다.  
이 프로젝트는 **그 지점을 잡아내기 위한 시스템**이다.

---

## 접근 방식

### Explicit Proxy (현재 PoC 방식)
- 클라이언트가 프록시의 존재를 알고 직접 요청을 보내는 방식 (브라우저 프록시 설정 / `curl -x`)
- 구조 검증에 적합, 빠르게 구현 가능

### Transparent Proxy (후속 확장 목표)
- 클라이언트는 프록시를 모르고 원 서버로 통신한다고 믿음
- 운영체제(커널) 레벨에서 트래픽을 프록시로 우회
- 실제 사내 배포에 필요한 방식이지만 커널 레벨 기술(WFP 등) 필요 → 로컬 PoC 완성 후 다음 단계

---

## 프록시 동작 구조

### Part 1 — 트래픽을 받아내는 방법

```
Client  ←→  Local Proxy  ←→  Origin Server
```

프록시는 **두 얼굴**을 가진다:
- 클라이언트에게는 **서버처럼** 응답
- 원 서버에게는 **클라이언트처럼** 요청

한 요청의 처리 흐름:
1. 클라이언트로부터 요청을 받는다
2. 내용을 읽어낸다 (응용 계층 해석)
3. 원 서버로 보낸다
4. 응답을 받는다
5. 클라이언트에게 돌려준다

→ 클라이언트는 서버와 직접 통신했다고 생각하지만, 프록시가 중간에서 요청 내용을 전부 보고 있는 상태.

### Part 2 — 파일을 식별하는 방법

3단계 접근:
1. **1차 판별**: 요청이 파일 업로드인지 HTTP 헤더(`Content-Type: multipart/form-data` 등)로 분류
2. **파싱**: 요청 안을 열어 헤더와 파일 데이터를 분리 (`Content-Disposition`, 멀티파트 경계 처리)
3. **검증**: 순수 바이너리에서 매직넘버(파일 시그니처)로 실제 파일 형식 검증, 확장자 위조 탐지

현재 상태:
- 1단계(업로드 1차 분류) 완료 및 검증
- 2~3단계(파서 로직) 연구 중

---

## OSI 계층 관점

- **전송 계층 이하**: 통신은 그냥 바이트 흐름으로 보임
- **응용 계층**: "이건 파일 업로드 요청이다", "어떤 서비스로 가는 요청이다" 등 의미가 드러남
- 이 프록시는 **응용 계층까지 올라가서 요청의 의미를 해석**하는 프록시

## 마일스톤 로드맵 (10주 커리큘럼 기반)

| 마일스톤 | 내용 | 주차 | 상태 |
|----------|------|------|------|
| M0 | 학습용 turn-based relay | W1~W2 | ✅ 완료 |
| M1 | 양방향(full-duplex) 릴레이 | W3 | 진행 예정 |
| M2 | 견고한 HTTP 파싱 + 로깅 | W4 | - |
| M3 | HTTPS 터널 (CONNECT, raw 통과) | W3 연장 | - |
| M4 | TLS MITM 기초 (Root CA + 클라측 TLS 서버) | W5 | - |
| M5 | TLS MITM 완성 (SNI 동적 인증서 + 평문 분석) | W6 | - |
| M6 | 파일 업로드 행위 식별 ★핵심 | W7 | - |
| M7 | 파일 메타 + 바이너리 추출 | W8 | - |
| M8 | 파일 분석 + PoC 검증 (다환경 테스트) | W9 | - |
| M9 | 최종 보고서 + 발표 | W10 | - |

### 마일스톤별 핵심

**M1 — 양방향 릴레이**
- `relay(from, to)` + 동시화(스레드 2개 또는 WSAPoll)
- 완료기준: 큰 응답 + keep-alive에서 데드락/누수 없이 양방향 동작, 한쪽 종료 시 반대쪽도 종료

**M2 — 견고한 HTTP 파싱**
- `\r\n\r\n` 경계, 부분수신 누적, Content-Length/chunked body, keep-alive 루프
- 완료기준: 실제 브라우저/curl의 분할·keep-alive·chunked 요청 안정 파싱 + 로그

**M3 — HTTPS 터널**
- `CONNECT host:443` → `200 Connection Established` → 암호 바이트 양방향 릴레이
- 완료기준: 브라우저 프록시 설정 후 HTTPS 사이트 정상 로드

**M4 — TLS MITM 기초**
- Root CA 생성 + Windows 신뢰 저장소 등록, 고정 leaf 인증서로 클라측 TLS 서버 PoC
- 완료기준: 고정 leaf로 클라와 TLS handshake 성공, 인증서 신뢰됨

**M5 — TLS MITM 완성**
- ClientHello SNI 추출 → host용 leaf 인증서 동적 생성+캐시 → 평문 분석 → 재암호화 전달
- 완료기준: 핀닝 없는 HTTPS에서 평문 요청/응답 확인 + 재암호화 정상 전달

**M6 — 파일 업로드 식별 (★본 과제 핵심)**
- `multipart/form-data` 판별, `Content-Disposition` 파싱(filename), Content-Type, 파일 크기 추정
- 타깃 AI 서비스(ChatGPT/Gemini/네이버)별 업로드 지문 표 작성
- 완료기준: 타깃 2~3개 서비스의 업로드 식별 + 지문 표 정리

**M7 — 바이너리 추출**
- 멀티파트 본문 경계 분리 → binary 추출 → 해시 비교(원본 vs 추출본)
- 완료기준: 추출 파일이 원본과 바이트/해시 일치

**M8 — 파일 분석 + PoC 검증**
- 매직넘버(시그니처) 확인, 확장자 vs 실제 유형 불일치 탐지, 다환경 테스트
- 완료기준: 다환경 테스트 결과표(성공/실패/원인) 완성

## 진행 원칙

- **[직접]** 개념 핵심(중계·파싱·MITM 흐름·업로드 식별)은 직접 구현하며 이해
- **[AI]** 보일러플레이트/에러처리 골격/인증서·TLS API 호출부는 AI 활용 ("설명 가능"이면 됨)
- 매 단계: 동작 산출물 + 설명 한 줄 + 트레이드오프 메모
- 안 되는 케이스(한계)도 반드시 기록

## 현재 진행 중 & 후속 과제

**진행 중:**

### 슬라이드 4 — Explicit vs Transparent Proxy

- Explicit: 클라이언트가 프록시를 "인지"
- Transparent: 클라이언트는 프록시를 "모름"

Transparent는 커널 레벨 기술이 필요 → 후속 확장 과제.
현재 PoC는 Explicit 방식으로 구조 검증.

대본 핵심: 실제 보안 제품에서는 Transparent 방식을 씀. 프록시 설정을 무시하고 자체 통신하는 프로그램도 잡아야 하기 때문. 이번 PoC 범위에서는 Explicit으로 구조를 먼저 검증.

---

### 슬라이드 5 — Part 1: 트래픽 받아내기

프록시의 두 얼굴:
- 클라이언트에게는 서버처럼 응답
- 원 서버에게는 클라이언트처럼 요청

흐름: 요청을 받는다 → 내용을 읽어낸다 → 원 서버로 보낸다 → 응답을 받는다 → 클라이언트에게 돌려준다

검증: 실제 요청이 정상적으로 왕복하는 것을 확인.

---

### 슬라이드 6 — Part 2: 파일 식별

최종 목표: 브라우저가 AI 서비스로 파일을 업로드 → 프록시가 그 요청 안의 파일을 식별.

3단계:
1. 요청이 파일 업로드인지 1차 판별 → **완료**
2. 요청 안을 열어 헤더와 파일 데이터를 분리 → 연구 중
3. 순수 바이너리에서 실제 파일 형식을 검증 → 연구 중

서비스마다(ChatGPT, Claude, 일반 웹) 요청 구조가 다르므로 서비스별 패턴 분석 병행 중.
